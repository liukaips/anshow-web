# AnShow 对外 PPT 内容与官网体验升级设计

Date: 2026-07-18  
Status: Approved for implementation

## 1. 背景与目标

本设计以 `/Users/kai.liu/Downloads/AN-SHOW 对外.pptx` 为公司内容来源，升级现有 AnShow 官网的公开内容、页面信息层级、案例表达、图片匹配和运营后台体验。

本轮不是重新搭建网站，也不是替换已经交付的技术架构。现有 Next.js 前端、Hono 后端、Drizzle、Better Auth、SQLite、GSAP、Docker Compose 和 Caddy 自动 HTTPS 保持不变。改造重点是：

- 把 PPT 中的公司事实、业务能力、资质、项目案例和联系方式转化为适合官网阅读的三语内容；
- 将上述内容写入 SQLite，作为首次部署时直接可发布的基础数据；
- 保留现有深色首屏、品牌蓝、路线动效和深浅交替的科技风格，改善内容可信度与转化路径；
- 为缺少的项目场景补充可信的图片，并提供桌面、移动端和现代图片格式；
- 把 Admin 调整为适合非技术运营人员使用的全中文传统管理后台。

官网的第一目标是收集运输询价。企业资质、项目数据、服务流程和全球协同能力用于建立信任并支持询价决策。

## 2. 已确认决策

- 采用“证据驱动的全球物流”视觉方向，尽量保持现有官网风格，不做无必要的全站换皮。
- PPT 内容不机械照搬。保留事实与业务含义，改写为适合官网、自然且可举证的表达。
- 服务与资质可以按 PPT 的实际业务范围发布；案例统一标记为“代表性项目”，不披露客户名称。
- 弱化“100% 满意”“零违规”“全球无死角”等绝对承诺；保留可解释的货物、路线、尺寸、重量和时效数据。
- 英文为默认语言，中文和俄语提供完整内容。语言根据浏览器自动选择，用户可随时切换。
- PPT 内容必须进入数据库；初始化可重复执行，但不得覆盖 Admin 后续编辑。
- 图片优先使用现有合格素材与 PPT 中的真实公司图片。缺少项目场景时允许生成写实工业图片。
- Admin 参考用户提供的传统管理后台截图：深色侧栏、浅灰工作区、白色表格、低装饰、清晰子菜单。
- Admin 全中文。内容名称、字段名称和配置项必须与前台位置和业务含义直接对应，不向运营人员暴露 JSON、code、slug 或数据库字段。

## 7. 三语内容策略

- 英文面向海外客户和代理合作伙伴，强调服务范围、合规能力、项目数据和询价效率。
- 中文面向国内出口企业，使用自然货代术语，解释资料、运输方式和项目执行。
- 俄语面向中俄业务和俄语客户，使用自然俄语物流术语，不套用中文语序。
- 公司名称、UN 编号、尺寸、重量、时效和联系信息在三种语言中保持一致。
- 标题、按钮和摘要按语言重写，不做逐字直译。
- 某种语言尚未完成时，Admin 显示“待补充”，公开页面回退英文，并保留用户当前语言导航。
- 页面提供 canonical 与 `hreflang`。语言切换优先进入同一内容的对应翻译，没有对应内容时进入目标语言首页。

## 8. SQLite 内容模型

继续复用现有本地化内容表：`heroSlides`、`services`、`tradeLanes`、`cargoTypes`、`caseStudies`、`articles`、`certificates`、`proofMetrics`、`pages`、`navigationItems` 及其翻译表。

### 8.1 结构化详情正文

现有 `body` 为 `TEXT`。本轮在该字段中保存带版本号、经过 Zod 校验的 JSON，不新增富文本依赖：

```json
{
  "version": 1,
  "sections": [
    { "type": "paragraph", "text": "..." },
    { "type": "fact-list", "items": [{ "key": "weight", "label": "货重", "value": "12", "unit": "吨" }] },
    { "type": "process", "steps": [{ "title": "需求诊断", "text": "..." }] },
    { "type": "callout", "title": "准备资料", "text": "..." }
  ]
}
```

允许的区块类型固定为段落、事实列表、流程、要点列表、引用提示和询价提示。前端不执行数据库内容中的 HTML。

旧的纯文本 `body` 继续按普通段落渲染。结构化 JSON 解析失败时记录错误并回退为安全纯文本，避免旧内容或单条异常导致页面崩溃。

Admin 不显示 JSON。不同内容类型使用业务表单生成结构化数据：

- 服务：适用货物、服务范围、流程、准备资料；
- 案例：货物、起运地、目的地、挑战、方案、结果；
- 航线：区域、可用方式、适用货物、协同说明；
- 资质：资质名称、说明、编号、有效期、验证来源和证书图片；
- 页面：介绍、事实、联系信息和行动提示。

### 8.2 媒体关系

每条内容继续通过 `mediaId` 关联媒体。媒体记录包含：

- 原始存储键；
- 宽度、高度和主色；
- 视觉焦点；
- AVIF 和 WebP 衍生图；
- 三语替代文本；
- 使用位置。

首页首屏需要桌面和移动端独立构图时，使用同一媒体组中的明确变体，不直接拉伸横向图片。

## 9. 基础数据初始化

基础数据目录扩展为：

- 4 个首页轮播；
- 7 项服务；
- 4 组重点航线；
- 4 类特殊货物；
- 8 个代表性案例；
- 4 项资质；
- 公司与联系页面；
- 服务承诺和首页证明指标；
- 3 篇中性基础文章；
- 公开导航项目。

每条记录使用稳定 `code`。初始化在数据库迁移完成后运行，并使用单个事务和 `onConflictDoNothing`：

- 首次部署创建表并写入完整已发布基础数据；
- 后续部署补充不存在的记录和翻译；
- 已存在且由运营人员修改过的标题、正文、图片、排序和发布状态均不得被覆盖；
- 任一基础记录或结构化正文校验失败时整批回滚，并记录集合、code、语言和字段；
- 本地数据库、生产数据库和恢复数据库使用同一初始化入口。

当前环境已经存在旧版通用基础内容，因此仅使用 `onConflictDoNothing` 不能完成本次 PPT 内容升级。增加轻量的基础内容版本记录，保存集合、code、语言、已应用版本和已应用内容指纹：

- 当前内容与上一版基础内容指纹完全一致时，视为未被人工编辑，可以升级到 PPT 基础版本；
- 任一可见字段、媒体、排序或发布状态与上一版指纹不同，视为人工编辑，跳过升级并记录“保留运营修改”；
- 新 code 直接插入；
- 被新版目录替代的旧记录只有在仍与旧基础内容完全一致时才自动归档，人工修改过的旧记录继续保留；
- 升级完成后记录新版本和新指纹，后续部署继续使用相同判定；
- 基础版本状态不作为公开内容，不出现在前台 API。

例如旧版 `multimodal` 服务若仍是系统原始通用文案，可以归档并由新的“保险与定制方案”替代；若运营人员已经修改该服务，则保留该记录，并在 Admin 的升级结果中提示人工确认，不静默删除。

资质编号和证书图片可以为空。记录发布不等于凭证已验证；只有 `verifiedAt` 和 `verificationSource` 存在时，前端才显示“已核实”及凭证详情。

## 11. 图片性能

- 生成母图不少于 2048px；母图不直接提供给浏览器。
- 桌面输出 1920、1280、768 和 480 宽的 AVIF 与 WebP。
- 首屏和重点案例输出 768×1024 左右的移动端独立构图。
- 缩略图输出 320px AVIF。
- 所有图片声明宽高或 `aspect-ratio`，并记录主色和视觉焦点。
- 当前首屏第一张图片预加载；其他轮播和折叠以下图片延迟加载。
- 桌面首屏目标不超过 280KB，移动首屏目标不超过 140KB，普通内容图目标不超过 100KB。
- 图片缺失或加载失败时使用带品牌色、运输方式名称和固定宽高比的降级背景，不显示突兀灰白空轨道。
- 前端使用 `picture`、`srcset` 和 `sizes`，不得通过 CSS 拉伸图片。

## 12. 动效系统

继续使用现有 GSAP 和 ScrollTrigger，不新增动效依赖。

### 12.1 核心动效

- 首页轮播：图片交叉淡化、轻微平移、文字独立进入和可暂停进度；
- 路线推进：起运地、目的地和运输节点随滚动逐步激活；
- 五步流程：当前步骤点亮，路线与说明同步切换；
- 案例证据：重量、尺寸、时效等指标一次性揭示；
- 页面转场：250 至 400ms 的品牌路线覆盖转场；
- 列表出现：30 至 50ms 的轻量错峰，只在首次进入可视区域时执行。

### 12.2 性能与可访问性

- 普通动效只改变 `transform` 和 `opacity`；
- 不使用滚动劫持，不在动画期间阻止操作；
- 动画可被滚动、导航和触摸打断；
- 移动端减少视差层和粒子数量；
- `prefers-reduced-motion` 下立即显示最终状态，停止自动轮播、计数和路线动画；
- JavaScript 未加载时，页面仍显示完整可读内容。

## 13. Admin 运营后台调整

本节是 `2026-07-16-admin-operations-rebuild-design.md` 的视觉和信息架构补充。既有审核、预览、权限、询价、媒体、备份和审计能力继续保留。

### 13.1 视觉

- 深灰固定侧栏、白色顶栏、浅灰工作区和白色内容面板；
- 主操作使用 AnShow 品牌蓝；绿色只表示成功，黄色表示待处理，红色表示错误或危险；
- 圆角 4 至 6px，边框清楚，阴影克制；
- 表格使用传统表头、稳定行高、标准分页和文字操作；
- 不使用营销式大标题、渐变、玻璃效果、浮动装饰和花哨表格动画；
- 实现时使用现有 Lucide 图标，不使用字符或 Emoji 作为正式图标。

### 13.2 侧栏与子菜单

侧栏分为：

1. 日常工作
   - 工作台；
   - 询价管理：全部询价、待联系、跟进中、已完成。
2. 官网内容
   - 内容管理：首页内容、物流服务、特殊货物、重点航线、项目案例、行业洞察、关于与联系；
   - 图片管理；
   - 审核与发布：待审核内容、预约发布、发布记录。
3. 系统管理
   - 网站设置：公司与联系方式、语言与默认设置、询价通知、备份与恢复；
   - 员工与权限；
   - 操作记录。

仅有一个子菜单组默认展开：当前路由所属组。侧栏桌面可滚动，移动端变为抽屉。无权限项目不显示。

### 13.3 工作台

工作台展示系统真实数据：

- 待联系询价；
- 待审核内容；
- 本月新增询价；
- 上次自动备份时间和状态；
- 最近编辑的官网内容；
- 翻译缺失、超时询价和审核待办；
- 新增案例、修改轮播、上传图片和备份恢复等常用操作。

不显示无来源的销售额、订单额或虚假业务图表。

### 13.4 内容列表和编辑

默认表格列为：

- 内容标题；
- 前台位置；
- 中文、英文和俄语完成情况；
- 发布状态；
- 最后修改时间；
- 编辑、预览等操作。

运营人员看到的配置名称必须与前台一致，例如“首页首屏轮播”“项目案例”“公司与联系方式”。系统自动维护 code、slug、排序键、媒体尺寸和结构化正文，技术信息只在有权限的高级设置中只读显示。

编辑流程按业务类型展示字段。保存草稿后可以打开完整前台预览，确认桌面、移动端、三语和图片裁切后再提交审核或发布。

## 14. 组件边界

### 14.1 前端

- `HomepageSections`：组合首页数据，不包含数据库访问；
- `HeroCarousel`：轮播、暂停、进度、键盘和触摸控制；
- `ResponsiveContentMedia`：桌面和移动端媒体选择、占位与错误回退；
- `ProcessStory`：五步流程和减少动态效果；
- `StructuredContentRenderer`：仅渲染通过 Zod 校验的固定内容区块；
- `EvidenceMetric`：以统一格式显示重量、尺寸、时效和路线数据；
- `QuoteEntry`：首页简化询价和完整询价页入口；
- Admin 通用壳层、表格、字段、状态、翻译进度和预览组件继续复用既有后台组件。

### 14.2 后端

- 内容基础数据目录：定义稳定 code、三语翻译、结构化正文和媒体引用；
- 初始化运行器：迁移完成后事务化执行幂等初始化；
- 结构化内容校验器：在初始化、Admin 保存和公开读取边界校验；
- 公开内容仓库：只返回当前语言已发布内容和允许的英文回退；
- 媒体服务：生成衍生图、记录焦点和维护使用关系；
- 工作流服务：草稿、预览、审核、发布和回退；
- 站点设置：统一保存公司信息、联系信息和公开渠道。

各单元通过现有类型化 API 通信，不让前端组件直接解释数据库结构。

## 15. 数据流

### 15.1 首次部署

1. Docker 启动后端。
2. Drizzle 执行数据库迁移。
3. 初始化运行器校验完整基础数据目录。
4. 单个事务写入缺失的基础记录和三语翻译。
5. 媒体目录注册已存在的衍生图和使用关系。
6. 公开 API 返回可直接展示的已发布内容。

### 15.2 Admin 编辑与发布

1. 运营人员通过中文业务表单编辑内容。
2. 系统自动保存草稿并校验结构化正文、图片和三语完整度。
3. 用户生成完整前台预览快照。
4. 审核人员检查页面、翻译和移动端裁切。
5. 发布操作原子更新已发布版本并清除公开内容缓存。
6. 基础数据初始化永远不覆盖该记录。

### 15.3 公开读取

1. 根路径根据浏览器语言跳转。
2. Next.js 服务端通过类型化 API 获取已发布内容。
3. 结构化正文通过 Zod 校验后渲染。
4. 图片根据视口选择移动或桌面资源。
5. 缺少当前语言时回退英文；缺少图片时显示品牌降级背景。

## 16. 错误处理

- 初始化数据错误：事务回滚，启动日志输出集合、code、语言和字段，不留下半套数据。
- 结构化正文错误：Admin 阻止保存；公开页面对旧数据回退安全纯文本并记录请求 ID。
- API 不可用：保留页面骨架，显示当前语言的错误信息和重试操作。
- 内容不存在：显示可访问 404，并提供返回首页、服务和询价入口。
- 翻译缺失：Admin 显示明确待办；公开页面回退英文。
- 图片缺失或加载失败：使用固定宽高比的品牌降级背景，不显示空白、破图或灰白轨道。
- 图片生成不合格：不登记为已发布媒体，继续使用现有合格图片。
- 预览令牌失效：显示中文失效页，不回退线上内容。
- 发布失败：事务回滚，不允许出现部分语言或部分页面已更新。
- 自动保存失败：保留本地表单值，持续显示恢复办法和重试按钮。

## 17. 测试策略

### 17.1 后端

- 初始化可在空数据库上创建完整内容；
- 第二次初始化不产生重复记录；
- 人工修改标题、正文、图片和状态后再次初始化不被覆盖；
- 旧版通用基础数据库可把未修改记录升级为 PPT 版本；
- 旧版内容中被人工修改的记录在版本升级时继续保留；
- 被新版目录替代且未修改的旧基础记录可以归档，已修改记录不自动归档；
- 初始化器与生产迁移入口使用同一路径；
- 结构化正文三语校验和旧纯文本兼容；
- 8 个案例、7 项服务、4 类特殊货物、4 组航线和 4 项资质的公开 API 契约；
- 草稿、预览、审核和发布事务；
- 无效预览、媒体缺失和语言回退；
- 备份包含 SQLite 与上传媒体，恢复后再次初始化不覆盖内容。

### 17.2 前端和 Admin

- 首页各章节使用数据库内容，不依赖硬编码 PPT 文案；
- 轮播运输方式与图片正确匹配；
- 结构化服务和案例详情渲染；
- 图片 `srcset`、移动端构图、错误回退和布局稳定；
- `prefers-reduced-motion` 下关闭复杂动画；
- Admin 全中文扫描；
- 子菜单展开、权限可见性、传统表格筛选和分页；
- 业务表单不显示 JSON、code 或 slug；
- 三语完成状态、前台位置、草稿预览和发布流程。

### 17.3 E2E 与视觉验收

- 英文、中文和俄语首页、服务、案例、航线、关于、联系和询价关键路径；
- Chrome 与 Safari；
- 375、768、1024 和 1440 像素宽度；
- 无横向滚动、文字遮挡、异常大留白、灰白空轨道和图片变形；
- 所有基础媒体 URL 返回 200；
- 首屏只预加载当前图片，折叠以下图片延迟加载；
- 键盘导航、焦点、图片替代文本和 WCAG AA 对比度；
- Admin 创建案例、三语确认、前台预览、提交审核和发布完整流程。

最终验证执行前后端 lint、类型检查、单元测试、集成测试、生产构建和 Playwright 关键路径。

## 18. 验收标准

- 新数据库部署后无需人工录入即可展示完整三语官网和图片。
- PPT 中的公司事实、服务、资质、八个案例和联系方式都能在公开页面找到，并采用已确认的保守宣传口径。
- 所有公开内容来自 SQLite；前端只保留界面标签和通用交互文案。
- Admin 修改内容后重新部署和初始化不会被覆盖。
- 首页以询价为主目标，深色科技首屏与明亮内容章节保持现有品牌连续性。
- 动效解释运输路线、步骤和案例，不造成滚动阻塞或移动端卡顿。
- 图片与运输方式、货物类型正确匹配，并提供桌面和移动端优化版本。
- Admin 符合参考图的传统 B 端风格，菜单分组和子菜单清楚，表格简洁。
- 非技术人员无需理解内部编码、JSON 或数据库字段即可编辑、预览和发布前台内容。
- Chrome 与 Safari 在目标宽度下无大块异常留白、未对齐、图片变形或不可操作控件。
- 全部测试、类型检查、lint 和生产构建通过。

## 19. 与既有设计的关系

本设计在以下已批准设计之上实施：

- `2026-07-14-anshow-logistics-website-design.md`：总体产品、技术与部署架构；
